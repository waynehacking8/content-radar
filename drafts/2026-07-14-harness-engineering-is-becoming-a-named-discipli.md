There's now a curated list on GitHub for "harness engineering" — tools, patterns, evals, memory, permissions, observability, and orchestration for AI agents. A year ago that wouldn't have been a category. Now it's specific enough to have its own awesome-list.

That's a useful signal for anyone tracking where the actual work is moving. The model is increasingly the boring, commodity part. The hard part — the part that decides whether an agent is trustworthy enough to run against production code or customer data — is the harness around it: what tools it can call, what it's allowed to touch, how its memory persists, how you observe what it did after the fact.

That's not a research problem, it's an integration and judgment problem, and it looks a lot like the job description for solutions architects and forward-deployed engineers who have to make agents work inside one specific company's constraints rather than in a demo. If "harness engineering" is becoming a named discipline, that's a good sign the skill set is being recognized, not just improvised.
